# 北京交通大学就餐仿真系统设计规格说明书

## 1. 系统总体目标
北京交通大学就餐仿真系统面向校园食堂就餐场景，构建“用户端点餐预约 + 管理端核销监管”的完整闭环。系统目标是：
1. 提供便捷的线上点餐、下单、预约和取餐流程，减少线下排队等待。
2. 支持食堂就餐容量与时段管理，实现预约分流与有序就餐。
3. 为管理人员提供实时看板和历史统计，辅助运营决策。

依据项目现有实现，系统已覆盖用户注册登录、浏览菜品、购物车、下单支付回调、座位预约、二维码取餐、管理员扫码核销、统计看板等核心流程（参考：`README.md`、`tests/full_link_test.py`、`docs/api/backend-modules.md`）。

## 2. 系统开发架构（B/S 架构）
本系统采用 **B/S（Browser/Server）架构**。

- 浏览器端：Vue 3 单页应用（用户端与管理端页面）。
- 服务端：Spring Boot 提供 REST API。
- 数据层：MySQL 8 存储业务数据。

交互方式：前端通过 HTTP/JSON 调用后端接口（`/api/v1/user/**`、`/api/v1/admin/**`）。

## 4. 系统架构

### 4.1 模块划分
系统分为 6 个核心模块：

1. **认证与权限模块**
   - 功能：用户注册、登录、JWT 发放与校验、请求身份识别。
   - 相关接口：`/api/v1/user/register`、`/api/v1/user/login`
   - 协同：为用户端和管理端接口提供统一身份认证。

2. **用户点餐模块**
   - 功能：食堂列表、菜品浏览、购物车管理、订单创建与订单详情。
   - 相关接口：`/api/v1/user/canteens`、`/api/v1/user/cart`、`/api/v1/user/orders`
   - 协同：订单创建读取购物车数据并生成订单明细。

3. **预约与取餐码模块**
   - 功能：时段+座位预约、取餐二维码生成、二维码状态管理。
   - 相关接口：`/api/v1/user/reservations`、`/api/v1/user/pickup-qrcodes`
   - 协同：依赖已支付订单，生成可核销二维码。

4. **管理核销模块**
   - 功能：扫码核销、核销结果反馈、核销日志记录。
   - 相关接口：`/api/v1/admin/pickup-qrcodes/scan`
   - 协同：消费二维码并回写状态，保证一次性核销。

5. **看板与统计模块**
   - 功能：实时人数看板、历史用餐人数统计、菜品销量统计。
   - 相关接口：`/api/v1/admin/canteens/realtime-occupancy`、`/api/v1/admin/statistics/*`
   - 协同：从订单、预约、核销等数据中聚合统计结果。

6. **基础数据管理模块**
   - 功能：食堂信息、时间段、座位、菜品的维护与查询。
   - 相关接口：`/api/v1/admin/canteens/{id}/foods` 等
   - 协同：为下单、预约、统计提供主数据支持。

### 4.2 模块协同与接口形式
- 接口形式：RESTful + JSON，统一响应结构 `{code, message, data}`。
- 典型协同链路：
  1. 用户登录获取 token；
  2. 浏览菜品并加入购物车；
  3. 创建订单并支付回调更新状态；
  4. 基于已支付订单预约座位；
  5. 生成二维码；
  6. 管理员扫码核销并记录日志。

### 4.3 模块间交换数据
- 认证模块输出：用户身份（userId/role/token）
- 点餐模块输出：订单号、订单金额、订单状态、订单明细
- 预约模块输出：reservationId、时段、座位、预约状态
- 二维码模块输出：qrToken、expiresAt、qrStatus
- 管理模块输出：核销结果、实时占用率、历史统计数据

## 5. 软件原型

### 5.1 页面清单（依据前端路由）

#### 用户端页面
1. `/user/login` 登录页：手机号+密码登录。
2. `/user/register` 注册页：新用户注册。
3. `/user/canteens` 食堂列表页：查看食堂及预约人数。
4. `/user/canteens/:id/foods` 菜品页：浏览菜品、加入购物车。
5. `/user/cart` 购物车页：调整数量、删除、提交订单。
6. `/user/orders/:id` 订单详情页：查看订单、预约座位。
7. `/user/qrcode` 取餐码页：查看二维码与状态。

#### 管理端页面
8. `/admin/dashboard` 实时看板页：查看各食堂就餐占用情况。
9. `/admin/verify` 扫码核销页：扫描或输入二维码令牌核销。
10. `/admin/foods` 菜品管理页：菜品增删改查与上下架。
11. `/admin/statistics` 统计报表页：人数与销量趋势分析。

### 5.2 页面跳转关系
- 用户主流程：登录/注册 → 食堂列表 → 菜品页 → 购物车 → 订单详情（可预约）→ 取餐码。
- 管理主流程：登录后进入管理布局 → 看板/核销/菜品管理/统计页面间切换。
- 未登录访问受限页面时，路由守卫自动跳转至 `/user/login`。

## 6. 数据模型

### 6.1 核心数据实体（数据库 11 张核心表）
- 用户：`users`
- 食堂：`canteens`
- 时间段：`canteen_time_slots`
- 座位：`seats`
- 菜品：`food_items`
- 购物车：`shopping_cart_items`
- 订单主表：`orders`
- 订单明细：`order_items`
- 预约：`reservations`
- 取餐二维码：`pickup_qr_codes`
- 核销日志：`qr_scan_logs`

### 6.2 数据产生与流转
1. **用户数据**：在注册时写入 `users`，登录时参与鉴权。
2. **购物车数据**：用户选餐后写入 `shopping_cart_items`。
3. **订单数据**：下单时创建 `orders` 与 `order_items`，并清空购物车。
4. **支付状态数据**：支付回调更新 `orders.order_status` 为已支付。
5. **预约数据**：基于已支付订单写入 `reservations`。
6. **二维码数据**：生成取餐码写入 `pickup_qr_codes`。
7. **核销数据**：管理员核销后写入 `qr_scan_logs` 并更新二维码状态。

### 6.3 数据存储方式
- 持久化存储：MySQL（InnoDB + 外键约束 + 索引）。
- 访问层：MyBatis Mapper + XML。
- 接口传输：JSON DTO。
