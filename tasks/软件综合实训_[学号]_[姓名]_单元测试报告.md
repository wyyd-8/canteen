## 1. 测试什么？
本次测试主要针对**智慧校园食堂系统核心业务链路的 API 接口**，测试内容包含：

- **函数与模块交互输入输出的正确性**：验证系统在前后端交互中，各个 Controller 层接口是否能正确解析前端发送的 JSON 请求载荷，同时保证后端返回的数据结构能够按照统一结果封装、输出正确的 JSON 结构。
- **核心业务逻辑与流程流转的正确性**：检验系统整个全链路点餐预约流程的状态流转是否正确。测试模块包括但不限于：
  1. 用户模块：用户注册、密码验证和用户账号登录（JWT Token 生成正确性）。
  2. 基础数据展示：查看公开的食堂列表及该食堂下的菜品列表信息。
  3. 购物车及订单管理工作流：添加菜品至购物车、创建订单、模拟微信支付回调修改订单状态为"已支付"。
  4. 预约与排队：校验可预约的营业时间段及座位余量查询逻辑，并验证用户绑定订单、时段、空闲座位生成预约记录（Reservation）的功能正确性。
  5. 扫码核销检验：基于用户订单及预约信息生成独有的"取餐核销二维码"，并检验管理端操作员是否能正常完成扫描核销流程操作。

---

## 2. 如何测试？
本次测试采用了**自动化测试脚本驱动**机制而不是人工手动操作前端页面。

- **测试方式**：通过编写基于 Python requests 库的全流程接口串联自动化脚本（tests/full_link_test.py 及 tests/test_time_slot_init.py）进行系统集成与单元化逻辑链路测试。
- **输入数据**：
  - **用户业务流**：动态生成满足符合手机号位数的虚拟手机号码（如通过挂载时间戳截取避免重复）与默认密码 password123 作为输入提交注册并随后进行登录。
  - **订单与预约流**：固定获取已有的 canteenId 与 foodId 传入参数，购物车请求输入具体数量（例如 quantity: 2）；发起预约时请求携带有效 orderId，系统根据当前日期读取可用营业时间段（如 11:00-12:00 阶段），在查询到空余的有效 seatId 组合成 JSON 体发起请求。
- **预期输出**：
  - 后端在接收正确参数后，响应必须符合 {"code":0, "message":"ok", "data": {...}} 结构；
  - 核心接口的响应状态 HTTP Code 应为 200；
  - 能够正确流转订单及核销状态，能获取带签名效力的 TOKEN 及一次性 qrToken 二维码核销标识；
  - 系统成功返回对应环节被生成的自增 ID 及相关的快照内容。
- **判定标准**：
  - 代码中使用严格的 assert 定言（如 assert resp.status_code == 200、assert len(slots) > 0）；如有任何环节流程产生错误状态码或抛出参数校验等异常即判定流程错误且测试不通过。脚本无抛错成功运行走完全链路则判定业务逻辑正确。

## 3. 实际测试及测试结果是什么？
- **实际测试情况**：使用 Python3 执行了本地运行服务下的集成链路测试脚本。脚本依序完成了注册（获取 userId=3）、登录带出 USER_TOKEN，并在加入菜品后创建了一笔30.00元的单据；完成支付回调通知后锁定系统当天 11:00-12:00 时段以及 A-001 号座位并获取核销令牌，最后交由管理端接口确认核销成功。
- **测试结果**：**测试顺利通过全链路各项断言，无异常抛出。各业务模块交互与数据流转状态符合预期设定。**
