## 1. 在本阶段中个人遇到了哪些问题，如何解决的

在系统的开发阶段，由于前后端分离架构的复杂性以及业务逻辑的严谨性要求，我主要遇到了以下几个典型问题：

### 问题一：前后端联调时的跨域（CORS）与状态保持问题
**问题描述**：在本地开发时，前端（Vue3 + Vite, 运行在5173端口）向后端（Spring Boot, 运行在8080端口）发起API请求时，浏览器报跨域拦截错误。此外，登录后获取的 JWT Token 在后续请求中未能正确携带，导致鉴权失败。
**解决过程**：
1. **跨域问题**：在后端配置了全局的 CORS 映射，通过实现 `WebMvcConfigurer` 接口重写 `addCorsMappings` 方法，允许了前端域名的访问，并允许携带 Credentials。
2. **状态保持**：在前端封装了 `axios` 实例（`utils/request.js`），在请求拦截器（Request Interceptor）中统一从 localStorage 读取 Token 并附加到请求头的 `Authorization` 字段中；在响应拦截器中统一处理了 401 状态码，使其在 Token 失效时自动重定向至登录页。

### 问题二：高并发场景下的库存（菜品余量）超卖问题
**问题描述**：在测试“提交订单”接口时发现，如果通过脚本（如并发执行 `full_link_test.py`）模拟多个用户同时对同一份菜品下单，会出现菜品库存由于脏读而变成负数的情况。
**解决过程**：
为了解决并发一致性问题，我查阅了数据库事务隔离级别的相关资料，并结合业务实际情况，采用了**乐观锁**的机制。在 `FoodItemMapper.xml` 对应的扣减库存 SQL 中，加入了库存大于等于扣减数量的条件判断（`UPDATE food_item SET stock = stock - #{num} WHERE id = #{id} AND stock >= #{num}`），如果受影响行数为0，则抛出自定义的库存不足异常，并在 Service 层通过 `@Transactional` 注解保证订单和库存操作的原子性。

### 问题三：取餐二维码的安全性与时效性问题
**问题描述**：早期设计取餐二维码时，仅将订单号直接编码进入二维码中。这种做法存在极大的安全隐患，容易被恶意伪造和重放攻击。
**解决过程**：
重新设计了取餐码的生成机制。引入了动态 Token 和有效期限制。后端生成二维码数据时，结合“订单ID + 用户ID + 时间戳 + 随机盐”进行加密签名，并将对应的扫描状态存入 `PickupQrCode` 相关的表中。在后台管理端（`ScanVerify.vue`）扫描核验时，后端不仅会校验签名的合法性，还会检查二维码是否过期以及是否已被使用，极大提升了核销环节的安全性，并记录至 `QrScanLogMapper` 中便于回溯。

## 2. 在本阶段中个人承担了哪些任务，完成情况如何，过程中是否遇到问题，如何解决的

在本项目的开发阶段，我主要承担了**订单模块、购物车模块以及部分前后端交互和界面开发**的任务。

### 任务一：购物车功能的全栈开发
- **完成情况**：100% 完成。
- **工作内容**：后端编写了 `ShoppingCartMapper` 及相关的 Controller、Service，实现了加入购物车、修改数量、清空购物车等接口。前端开发了 `Cart.vue` 视图界面。
- **过程中遇到的问题**：前端在计算购物车总价时，由于 JavaScript 的浮点数精度丢失问题，导致金额显示出现多位小数（如 `0.1 + 0.2 = 0.30000000000000004`）。
- **解决过程**：在前端引入了自定义的金额计算工具函数，将金额先乘以100转为整数进行加减运算，最后再除以100并保留两位小数格式化展示，确保了财务数据的显示准确性。

### 任务二：核心订单生成与状态扭转流程开发
- **完成情况**：100% 完成。
- **工作内容**：完成了 `OrderController` 及 `OrderMapper.xml`，实现从选菜结算到订单生成的复杂逻辑，包括校验时间段（对接 `CanteenTimeSlotMapper`）、校验预约记录（`ReservationMapper`）、生成订单明细（`OrderItemMapper`）等。前端实现了 `OrderDetails.vue` 页面的渲染。
- **过程中遇到的问题**：该流程涉及多张数据表的操作，前期测试时发现如果订单创建成功但明细插入失败，会导致产生脏数据。
- **解决过程**：严格梳理了 Spring 框架中数据库事务的传播行为。在核心创建订单的 Service 方法上添加了 `@Transactional(rollbackFor = Exception.class)`，确保任何一个环节抛出异常（如通过断言抛出的业务层异常），整个订单创建过程都会进行回滚。

### 任务三：扫码核销端的接口与页面集成
- **完成情况**：100% 完成。
- **工作内容**：开发了食堂管理员验证用户取餐二维码的功能。包含 `ScanVerify.vue` 页面逻辑，对接后端的扫码接口，并记录核销日志。
- **过程中遇到的问题**：由于不同手机的摄像头像素和对焦速度不同，使用前端组件库调用摄像头进行二维码解码时，识别率不稳定。
- **解决过程**：对前端扫码组件的配置进行了优化，调整了扫码帧率和防抖（Debounce）处理。当扫码成功后，立刻暂停扫码事件解析并显示 Loading 状态，等待后端接口返回核销成功或失败的结果后再提示管理员，避免了反复发起无效的网络请求。

### 总结
本阶段按计划完成了我所负责的所有前后端开发工作，代码已通过基本的功能测试和接口测试（使用提供的 `full_link_test.py` 等工具）。在实际的开发过程中，不仅提升了对 Spring Boot 原理和 Vue 组件化开发的理解，还通过解决跨域、事务、并发等问题，积累了宝贵的工程实践经验。
